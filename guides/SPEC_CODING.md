# Spec Coding + Harness 开发规范

> 版本: v1.0 | 创建: 2026-05-12
> 参考: 得物技术《基于 Harness + SDD + 多仓管理模式的 AI 全栈开发实践》
> 适用: 所有中大型需求（如影视森林管理端重构）

---

## 一、核心理念

### 1.1 什么是 Spec Coding

**不是**写一个 `SPEC.md` 就叫 Spec Coding。

Spec Coding = **先定义清楚要做什么，再让 AI 按规范实现**。

核心流程：
```
需求分析 → SDD 文档（proposal + spec + tasks）→ AI 按 SDD 生成代码 → 验证 → 归档
```

### 1.2 什么是 Harness

**Harness = 给 AI 一个"模仿对象"，而不是让它凭空创造。**

| 做法 | 效果 |
|------|------|
| ❌ "请实现一个 CRUD 接口" | AI 自由发挥，风格不一致，返工多 |
| ✅ "参照现有 xxx 模块的风格，实现 yyy" | AI 模仿已有实现，可用性高，返工少 |

### 1.3 为什么需要这套规范

之前的问题（血的教训）：
- 写了 SPEC.md 但只是问题清单，不是可执行的技术规格
- AI 生成的代码风格不一致、复用率低
- 前后端接口字段对不上
- 没有验证环节，部署后才发现问题
- 7 小时没有产出（因为没有明确的执行路径）

## 二、SDD 文档体系

### 2.1 文档结构

每个需求/功能点产出以下文档：

```
tasks/YYYY-MM/任务名-MMDD/
├── proposal.md      # 需求提案：要做什么、为什么做
├── spec.md          # 技术规格：怎么做、接口设计、数据模型
├── design.md        # 详细设计：类图、SQL、前后端分工（可选）
├── tasks.md         # 任务拆分：每个 task = 一个可执行的代码变更
├── LOG.md           # 执行日志
└── REVIEW.md        # 复盘总结（完成后）
```

### 2.2 proposal.md — 需求提案

**目的**: 用非技术语言描述清楚要做什么。

```markdown
# [功能名称] 需求提案

## 背景
为什么要做这个功能？解决什么问题？

## 目标
做完之后达到什么效果？

## 范围
包含什么 / 不包含什么

## 风险
可能遇到的坑、技术难点
```

### 2.3 spec.md — 技术规格

**目的**: AI 和人类都能据此直接写代码的技术蓝图。

```markdown
# [功能名称] 技术规格

## 数据模型
- 表结构设计（字段名、类型、约束）
- 与已有表的关系

## API 设计
### 接口清单
| 方法 | 路径 | 说明 | 请求体 | 响应体 |
|------|------|------|--------|--------|

### 接口详情
#### POST /api/xxx
- 请求体: { 字段说明 }
- 响应体: { 字段说明 }
- 错误码: xxx

## 前端设计
### 页面结构
- 路由: /xxx
- 组件树: Page > Header > Table > Modal

### 交互逻辑
- 表单校验规则
- 操作反馈（Toast/Dialog）
- 加载状态

## 后端设计
### 分层架构
- Controller → Service → Mapper
- 参照: [已有模块路径]

### 关键逻辑
- 业务规则
- 边界处理
- 错误处理
```

### 2.4 tasks.md — 任务拆分

**目的**: 把大需求拆成可独立执行的小任务。

```markdown
# 任务拆分

## 任务依赖图
T1（后端实体）→ T2（后端接口）→ T3（前端页面）→ T4（联调）

## T1: [任务名]
- **前置**: 无
- **内容**: 创建 xxx 实体类、Mapper、Service
- **参照**: 现有 xxx 模块
- **产出**: 文件路径列表
- **验证**: 编译通过

## T2: [任务名]
- **前置**: T1
- **内容**: 实现 xxx Controller
- **参照**: 现有 xxxController
- **产出**: 文件路径列表
- **验证**: curl 测试接口返回正确
```

## 三、Harness 实践规则

### 3.1 找模仿对象（必须）

**每个任务必须指定参照模块。**

```
✅ "参照 CrawlerScheduleServiceImpl 的分层结构，实现 ContentServiceImpl"
✅ "参照 crawler/page.tsx 的表格+弹窗模式，实现 content/page.tsx"
❌ "实现一个内容管理服务"（没有参照 = AI 自由发挥）
```

### 3.2 复用优先

- 能复用的组件、工具函数、数据结构，直接复用
- "抄一份改一改" > "用新方式写"

### 3.3 约束生成范围

在 SDD 中明确指定：
- 参考文件路径（`@path/to/file.tsx:53-58`）
- 参考接口定义
- 命名规范（与项目现有风格一致）
- 技术栈约束

## 四、执行流程

### 4.1 标准流程（中大型需求）

```
┌─────────────────────────────────────────────────┐
│  Phase 0: 调研                                    │
│  - 了解现状、梳理问题                              │
│  - 输出: EXPLORE.md（可选）                        │
├─────────────────────────────────────────────────┤
│  Phase 1: SDD 编写                                │
│  - proposal.md → spec.md → tasks.md              │
│  - 人工审核 SDD，确认后才开始编码                    │
├─────────────────────────────────────────────────┤
│  Phase 2: 按 tasks.md 逐个执行                    │
│  - 每个 task: 写代码 → 验证 → 标记完成              │
│  - 有参照模块，不凭空创造                           │
├─────────────────────────────────────────────────┤
│  Phase 3: 联调 + 验证                              │
│  - 前后端联调                                     │
│  - 功能验证                                       │
├─────────────────────────────────────────────────┤
│  Phase 4: 归档                                    │
│  - 代码提交 + 部署                                │
│  - REVIEW.md 夭盘                                │
│  - 更新记忆                                      │
└─────────────────────────────────────────────────┘
```

### 4.2 简化流程（小型需求/Bug 修复）

```
问题定位 → 方案（1-2 句话）→ 修复 → 验证 → 提交
```

不需要完整的 SDD 文档，但需要：
- 明确改什么
- 明确参照什么
- 明确怎么验证

### 4.3 流程检查清单

每个任务完成后，检查：

- [ ] 代码风格与项目一致？
- [ ] 复用了已有组件/工具？
- [ ] 接口字段前后端对齐？
- [ ] 编译/构建通过？
- [ ] 功能手动验证过？
- [ ] 代码已提交？
- [ ] 文档已更新？

## 五、前后端并行开发

### 5.1 接口契约先行

前后端并行的前提：**接口契约已对齐**。

```
SDD 阶段:
  后端 spec.md 定义接口 → 前端 spec.md 引用同一接口定义
  
开发阶段:
  后端 Agent 按 spec 实现接口
  前端 Agent 按 spec 调用接口（Mock 数据先跑通）
  
联调阶段:
  前端连接真实后端 → 端到端验证
```

### 5.2 三阶段验证

| 阶段 | 做什么 | 目的 |
|------|--------|------|
| 阶段 1 | 前端 + Mock 数据 | 验证 UI 逻辑 |
| 阶段 2 | 后端独立编译部署 | 验证接口正确性 |
| 阶段 3 | 前后端联调 | 端到端验证 |

## 六、Anti-Patterns（反模式）

### ❌ 反模式 1: 没有参照就写代码
```
❌ "实现一个爬虫管理页面"
✅ "参照 content/page.tsx 的表格+筛选+弹窗模式，实现 crawler/page.tsx"
```

### ❌ 反模式 2: SDD 太粗
```
❌ "做个 CRUD 接口"
✅ "POST /api/content/movie — 请求体: {title, year, genre, status} — 响应: Result<Movie>"
```

### ❌ 反模式 3: 跳过验证
```
❌ 写完代码直接部署
✅ 编译通过 → 单元测试 → 功能验证 → 部署 → 线上验证
```

### ❌ 反模式 4: 没有归档
```
❌ 做完就完了，不写 REVIEW
✅ 复盘：哪些做得好、哪些踩了坑、下次怎么改进
```

## 七、OpenClaw 中的实践

### 7.1 任务目录规范

```
tasks/YYYY-MM/任务名-MMDD/
├── SPEC.md          # 合并的 SDD 文档（proposal + spec + tasks 合一）
├── EXPLORE.md       # 调研报告（可选）
├── LOG.md           # 执行日志
├── REVIEW.md        # 复盘（完成后）
└── ISSUES-*.md      # 问题清单（可选）
```

### 7.2 SPEC.md 模板（合并版）

```markdown
# [任务名称] - SDD

> 创建: YYYY-MM-DD
> 状态: 进行中
> 参照模块: [已有模块路径]

## 一、需求（Proposal）
[要做什么、为什么做]

## 二、技术规格（Spec）
### 数据模型
### API 设计  
### 前端设计
### 后端设计

## 三、任务拆分（Tasks）
### T1: ...
### T2: ...

## 四、验收标准
[怎么算做完]
```

### 7.3 执行规则

1. **SDD 先行**: 中大型需求必须先写 SDD，人工确认后再编码
2. **Harness 约束**: 每个任务必须指定参照模块
3. **逐个执行**: 按 tasks.md 顺序执行，每个 task 完成后验证
4. **实时更新**: LOG.md 记录进度，SPEC.md 更新状态
5. **完成即归档**: 代码提交 + 部署 + REVIEW.md

### 7.4 心跳检查规则

定时任务检查：
1. 当前 task 是否完成？
2. 如果完成 → 执行下一个 task
3. 如果未完成 → 继续执行，不汇报"没有进展"
4. 如果卡住 → 如实汇报卡住原因，请求支援

**禁止**: "没有进展"的汇报。要么在推进，要么在等支援，要么已完成。

**心跳任务的核心原则**: 定时任务 = 自动工作，不是自动汇报。
- 有 task 待执行 → 直接执行，完成后汇报
- 所有 task 已完成 → 汇报完成，等待新指令
- 卡住需要支援 → 汇报卡住原因，请求支援
- 绝不复读："没有进展"不是合法输出

## 八、规范自动加载规则

**任何操作前，必须先查指南索引，加载对应规范。**

- Git push 前 → 必须读 `guides/DEPLOY_RULES.md` 第五节
- 部署操作前 → 必须读 `guides/DEPLOY_RULES.md`
- 任务执行前 → 必须读 `guides/TASK_RULES.md`
- 定时任务执行前 → 必须读 `guides/TASK_RULES.md` + `guides/SPEC_CODING.md`

**违反规范 = 工作无效，必须返工。**
